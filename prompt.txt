Necesito un sistema completo, escalable y profesional que emule ECUs de múltiples marcas de vehículos (Chevrolet/GM, Ford, Toyota, BMW, Volkswagen/Audi, Mercedes-Benz, Honda, Nissan, Hyundai/Kia) usando Raspberry Pi + MCP2515.

El sistema debe escribirse en C++17/20 (NO Python). Debe responder a TODOS los comandos de diagnóstico como si fuera un vehículo real, soportando TODOS los modos y servicios OBD2/UDS.

4. COMPATIBILIDAD CON FreeBUF y OpenCode
═══════════════════════════════════════════════════════════════

El núcleo en C++ expone una API REST y WebSocket. Esto permite que las herramientas en Python de FreeBUF/OpenCode (fuzzing, replay, etc.) se comuniquen con el emulador vía HTTP/WS. Se proporcionarán scripts de ejemplo en Python que llaman a la API del emulador C++.

Alternativamente, se puede compilar el emulador como biblioteca y crear bindings Python (pybind11) para que las herramientas de FreeBUF/OpenCode llamen directamente a funciones C++.

Requerimiento: Documentar ambos modos de integración.

6. CÓDIGO REQUERIDO (Estructura completa en C++)
═══════════════════════════════════════════════════════════════

Archivo principal: main.cpp
- Inicializa hilos (CAN RX/TX, simulación, API REST).
- Lee configuración (config.json).
- Selecciona el fabricante activo (puede cambiarse vía API).
- Enruta comandos CAN a la implementación específica de la marca.
- Registra métricas y logs.

Clase base: manufacturers/BaseManufacturer (abstracta)
- Métodos virtuales puros: handleCanFrame(const can_frame&), getDTCs(), clearDTCs(), startSecurityAccess(), etc.
- Implementación común de OBD2 estándar (modos 01-0A) usando tablas de PIDs.
- Gestión de sesiones UDS (ISO 14229) con temporizadores.
- Framework de seed/key (algoritmos configurables por marca).

Implementación por marca (8 pares .hpp/.cpp):
- ChevroletGM: hereda de BaseManufacturer, añade Mode 22 (200+ DIDs), security access GM.
- Ford: MS-CAN switching, PATS, Ford-specific PIDs.
- Toyota: Mode 21, Mode 27 Toyota, smart key.
- BMW: UDS completo, DCAN, respuesta a EDIABAS/INPA.
- Volkswagen: UDS VW, adaptation channels, long coding.
- Mercedes: Star Diagnostics, UDS Mercedes, DAS emulation.
- Honda: HDS protocol, immobilizer.
- Nissan: Consult II/III, NATS.

API REST (usando Crow / oat++):
- GET /vehicles → lista modelos (JSON)
- POST /select → selecciona marca/modelo (JSON)
- GET /current → estado actual (marca, parámetros simulados, DTCs activos)
- POST /inject_fault → inyecta DTC
- POST /clear_faults → borra DTCs
- POST /set_parameter → cambia valor de un sensor (ej. RPM, temp)
- GET /diagnostics → reporte completo (DTCs, freeze frames, readiness)
- POST /start_recording → comienza grabación de tráfico CAN a archivo
- POST /replay → reproduce tráfico grabado desde archivo
- GET /metrics → métricas Prometheus (exposición en /metrics)
- POST /fuzz_start → inicia fuzzing (envío aleatorio de tramas)
- GET /logs → descarga logs (spdlog)

WebSocket:
- Canal para enviar actualizaciones en tiempo real de sensores (RPM, velocidad, temperatura, etc.) a clientes web.

Base de datos:
- SQLite3 con las tablas especificadas.
- Inicialización con seed data para todas las marcas (vehículos, PIDs, DTCs, perfiles de simulación).
- Uso de SQLiteCpp o sqlite3 nativo con wrappers modernos.

Logging y métricas:
- spdlog (log rotativo, niveles: debug, info, error).
- prometheus-cpp (endpoint /metrics).

Build system:
- CMake con detección de dependencias: libsocketcan, sqlite3, Crow, nlohmann/json, spdlog, prometheus-cpp (opcional), pthread.

Instalación y despliegue:
- Script de instalación (bash): install_multi_emulator.sh
  - Compila el proyecto con CMake.
  - Configura SocketCAN (sudo ip link set can0 up type can bitrate 500000).
  - Crea servicio systemd para arranque automático.
  - Genera configuración inicial config.json.
